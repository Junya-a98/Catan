# 独立バリアント設計ロードマップ

> **状態:** 第4段階と `forecast_events` の `core_v2` 拡張まで完了しました。`standard`、7イベントの `forecast_events`、標準19タイル版の `frontier` をWeb開始画面から選択できます。次はフロンティアの外周付き拡張盤面、表示スケール、探索AIの強化です。`trade2` は未実装です。

## 目的

通常ルールを安定したまま保ちつつ、対局ごとに異なる遊び方を追加できる境界を作ります。既存の小規模な `HouseRules` とは分離し、開始時に次のいずれか1モードだけを選ぶ構成を想定します。

- `standard`: 現在の通常ルール。バリアント処理を行わない
- `forecast_events`: 数ターン先の全体イベントを予告してから解決する
- `frontier`: 未探索地形を街道の到達に応じて公開する
- `trade2`: 常設市場、競売、条件付き契約を段階的に追加する

初期段階ではモード同士を組み合わせません。効果の組み合わせ、非公開情報、AI評価を個別に検証してから、将来の併用可否を判断します。

## 通常ルールとの互換境界

予定する設定documentは `kind`、`version`、モード固有の `options` を持ちます。設定がない既存セーブ・リプレイは `standard` として扱います。

```json
{
  "kind": "standard",
  "version": 1,
  "options": {}
}
```

`standard` は無処理の実装にし、現在の盤面生成、資源保存則、合法手、AI、セーブ、リプレイ、LAN / Webスナップショットを変えないことを回帰テストで保証します。バリアント固有の可変状態は通常の `phase` や `house_rules` へ混ぜず、独立した `variant_state` として保存します。

完全なセーブの `variant_state` は `format`、`version`、`kind`、設定の `config_fingerprint`、`public`、`private` を持ちます。LAN / Webスナップショットでは同じ文書から `private` キー自体を除去し、空の秘密領域さえ送信しません。受信側は公開文書の項目、version、kind、fingerprintを `rules.variant` と照合し、観戦履歴の保存境界でも `private` の混入を拒否します。設定または実行時状態がない旧データだけは、空の `standard` として復元します。

対局identityには設定versionとfingerprintを含め、異なるモードのセーブ・リプレイ・通信状態を取り違えないようにします。権威サーバーがすべての効果を解決し、クライアントは公開済みの状態と合法なsemantic commandだけを受け取る方針は維持します。

## `forecast_events`: 予告付き全体イベント

イベントは安定したIDを持つ宣言的なcatalogとして定義し、実行コードやcallbackを保存データへ入れません。イベントを予告する手番と解決する手番は絶対手番番号で記録し、実時間timerには依存させません。

互換catalog `core_v1` は既存セーブ・リプレイ向けに保存形式と並び順を固定し、次の2種類だけを含みます。

- `wheat_harvest_v1`（豊作）: 次に麦が通常生産された後、銀行に対象者全員分の余裕がある場合だけ各対象者へ1枚追加する。追加分を配れない場合も通常生産は取り消さない
- `sheep_drought_v1`（大干ばつ）: 発動から参加人数分の手番が完了するまで、羊タイルの生産を止める

新規対局の既定catalog `core_v2` は、この2種類を含む次の7種類を各1枚ずつ持ち、一部のイベントには予告対象parameterがあります。

| イベント | 予告と効果 | 終了条件 |
| --- | --- | --- |
| 豊作 | 次に麦が通常生産された後、銀行に対象者全員分の余裕があれば各対象者へ麦を1枚追加。追加できなくても通常生産は維持 | 次の麦生産で消費 |
| 大干ばつ | 羊タイルの生産を停止 | 発動から全員が1手番を完了 |
| 港湾封鎖 | 予告された1か所の交換所を使用不可にする。港を使わない銀行交易は可能 | 発動から2手番を完了 |
| 建設ブーム | 全員共通で、次に有料建設される街道1本の木または土1枚を免除。街道建設カードの無料街道では消費しない | 次の有料街道建設で消費 |
| 商人祭 | 成立した国内交易の双方へ、銀行在庫を各1枚ずつ減らしてランダム資源を支給。双方分の在庫がなければ追加配布しない | 発動から全員が1手番を完了 |
| 山賊襲来 | 予告数字の公開タイルから建物による生産力が最も高い場所へ盗賊を即時移動。捨て札と略奪は行わない | 発動時に即時解決 |
| 地震 | 予告方角の辺では新しい街道を建設できず、既存街道も接続と最長交易路の計算に使えない。街道コマは残す | 発動から全員が1手番を完了 |

`core_v2` の対象parameterは権威サーバーが非公開seedから決定し、イベントと同時に事前公開します。港湾封鎖はstableな `harbor_id`、山賊襲来は `5 / 6 / 8 / 9` のいずれか、地震は盤面を6方角に分けたsectorを使用します。保存・通信境界ではイベントIDとparameterの組み合わせを厳密に検証し、別catalogの形式を混ぜた状態は拒否します。

初回は2手番前に予告し、以後は6手番間隔で発動します。絶対完了手番で管理するため、AI速度、再接続、リプレイ再生速度には依存しません。予告・発動・終了をイベントログへ記録し、Webでは次のイベント、残り手番、現在有効な効果を常時確認できます。AIも公開された予告と有効効果だけを判断材料にします。

`core_v1` は従来どおり同じイベントが連続しない4枚の均等なbagを維持します。`core_v2` は7種類を1周期に各1回ずつ発動し、周期境界でも同じイベントが連続しません。未消費の同種効果が次の周期まで残った場合はスタックさせず発動時点だけを更新します。完全セーブ、再接続、リプレイは設定のcatalog ID、対象parameter、進行位置を復元するため、`core_v1` を暗黙に `core_v2` へ変換しません。

未来のイベントデッキと順番は権威サーバーの完全なセーブだけに保持します。LAN / Webの公開スナップショット、観戦、閲覧者別リプレイへは、予告済みイベントと対象、現在有効な効果だけを含めます。予告・発動・終了は重要イベントとして記録し、リザルトから対応するネットワークリプレイへ移動できます。

Web版をバリアントの主導UIとし、広い画面では詳細カード、狭い画面では盤面直前のcompactな予告帯を表示します。封鎖中の交換所と地震中の街道区画は盤面manifestにも状態を持たせ、ラベルだけでなく盤面上でも識別できます。AIは未来のdeckを見ず、公開済みの残り手番と対象だけから資源価値、街道割引、国内交易、港、盗賊、地震区画の評価を調整します。

## `frontier`: フロンティア探索

対局中にタイルや辺を追加するとstable IDが変わり、操作対象、再接続、リプレイが壊れる可能性があります。そのため、全地形と接続関係を対局開始時に事前生成し、一部の地形だけを霧で隠します。

未探索タイルの公開スナップショットでは、資源、数字、港などの秘密を伏せます。街道が探索境界へ到達したときに権威サーバーがタイルを公開し、公開イベントとして保存します。初期配置、生産、盗賊、港、合法な建設先、AI評価は、公開状態を共通の判定関数から参照します。

標準19タイル版では中央7タイルと初期盗賊タイルを公開し、公開領域に接する街道が触れた未探索タイルを公開します。未公開タイルは `UNKNOWN` として資源・数字・盗賊をマスクし、港はmanifestから除外します。ロビー作成時に権威サーバーが提出Seedを秘密Seedへ置き換え、公開ロビー・snapshot・manifestではSeedを0に固定するため、参加者は盤面生成を逆算できません。

隠れた地形は生産、初期資源、盗賊候補、港交易、建設候補、AIの生産・港評価から除外します。AIは公開情報だけで評価しつつ、未知タイルを開く街道に探索価値を加えます。Webでは生成した霧テクスチャ、公開数カード、公開時アニメーションを表示します。次にstable IDを対局開始時に確定できる外周付き盤面と、デスクトップ / Web双方の拡大縮小へ進みます。

## `trade2`: 交易2.0

交易2.0は一度に実装せず、次の順に進めます。

1. 資源を `available` / `reserved` / `total` で管理するresource ledger
2. 出品や入札資源を安全に拘束するescrow
3. 手番プレイヤーが出品・購入・取消できる常設市場
4. 手番外の応札権限を備えた公開競売
5. 期限と発動条件を型付きで限定した条件付き契約

resource ledgerとescrowを用意する前に資源を手札から単純に取り除くと、資源カードの保存則、7の捨て札、独占カードなどに不整合や回避手段が生じます。予約済み資源をどの処理で手札として数えるかを先に定義し、銀行・全プレイヤー・escrowを含む総数を検証します。

現在の通信は基本的に手番プレイヤーだけがcommandを送る前提です。競売以降はcommandごとの操作可能席とrevision競合を検証する必要があります。初期の常設市場は自分の手番中だけ操作可能にし、この権限モデルを変更せず導入します。

条件付き契約では任意コードや自由記述の式を実行せず、`turn_start` や公開された出目など、有限で検証可能な条件だけを利用します。

各段階でローカル対局、セーブ往復、権威サーバーのrollback、閲覧者別マスキング、リプレイ再現性、AIの合法手を確認します。特に秘密状態は、現在画面だけでなく過去のネットワークリプレイにも漏れないことを検証します。

## 主なリスク

- 予告イベントの効果を増やしすぎることによる組み合わせ爆発
- フロンティアの隠し地形が通信・観戦・リプレイへ漏れること
- 動的な盤面変更によるstable IDの不一致
- escrowと7の捨て札・独占カード間での資源保存則違反
- 手番外入札の二重操作、古いrevision、応答待ちによる進行停止
- AIが公開されていない地形、注文、入札を判断に使用すること

予告イベントはcatalogを分けて互換性と組み合わせを制御しました。次の実装単位はフロンティアの外周付き拡張盤面と表示スケールとし、その後にresource ledger / escrowを先行させた `trade2` へ進みます。
